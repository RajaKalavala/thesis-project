# Golden Dataset — Schema

Field-level reference for `data/processed/medqa_ragas_golden.jsonl`. Each line is a single JSON object representing one accepted golden sample.

---

## Final schema (per row)

| Key | Type | Source stage | Purpose |
|---|---|---|---|
| `golden_id` | str | Stage 1 (sampling) | Unique ID, e.g. `medqa_gold_001` |
| `question_id` | str | MedQA | Original MedQA question identifier (preserves provenance) |
| `question` | str | MedQA | Clinical question text |
| `options` | dict | MedQA | `{"A": ..., "B": ..., "C": ..., "D": ...}` |
| `correct_answer` | str | MedQA | Gold answer text (e.g. `"Ceftriaxone"`) |
| `answer_idx` | str | MedQA | Letter of correct option |
| `meta_info` | str | MedQA | `step1` / `step2&3` |
| `metamap_phrases` | list[str] | MedQA | Pre-extracted clinical concepts |
| `gold_context` | str | Pass 1 | Verbatim text of the 1–3 strongest evidence passages, concatenated |
| `gold_chunks` | list[dict] | Pass 1 | Per-chunk: `{chunk_id, book_name, support_level, reason}` |
| `evidence_keywords` | list[str] | Pass 1 | Medical terms that should appear in the supporting evidence |
| `reference_answer` | str | Pass 2 | Single-sentence ideal answer — used as RAGAS `ground_truth` |
| `reference_explanation` | str | Pass 2 | 3–6 sentence evidence-grounded explanation |
| `why_other_options_are_less_suitable` | str | Pass 2 | 1–3 sentences justifying the distractors are wrong |
| `hallucination_check_points` | list[str] | Pass 2 | Atomic claims a faithful generation must cover |
| `question_type` | str | Pass 2 | `diagnosis` / `treatment` / `mechanism` / `management` / `other` |
| `requires_multihop` | str | Pass 2 | `yes` / `no` — used for stratified architecture comparison |
| `validation_scores` | dict | Pass 3 | `{evidence_relevance_score, faithfulness_score, explanation_quality_score, hallucination_risk}` |
| `audit_issues` | list[str] | Stage 7 | Empty for accepted rows; populated for needs_review |
| `quality_status` | str | Stage 7 | `accepted` (only this in the final file) / `needs_review` / `rejected` |

---

## Mapping to RAGAS at evaluation time

When Notebook 05 (pending) runs RAGAS on a RAG architecture's outputs, it consumes only a subset of these fields:

| RAGAS field | Sourced from this dataset |
|---|---|
| `question` | `question` |
| `ground_truth` | `reference_answer` (or `reference_explanation` for the longer-form variant) |
| `contexts` | retrieved by the RAG system at eval time, **not** from this file |
| `answer` | generated by the RAG system at eval time, **not** from this file |

The remaining fields (`gold_context`, `gold_chunks`, `evidence_keywords`, `hallucination_check_points`, `question_type`, `requires_multihop`) are not consumed by RAGAS itself but enable thesis-specific analysis:

| Use case | Fields used |
|---|---|
| **Per-architecture retrieval diagnostics** — does Hybrid RAG retrieve the same chunks the gold says are needed? | `gold_chunks.chunk_id` ↔ retrieved chunk IDs from each architecture (compute precision/recall@k against gold) |
| **Stratified analysis — multi-hop fairness** — does Multi-Hop RAG actually win on multi-hop questions? | `requires_multihop` |
| **Hallucination scoring beyond RAGAS faithfulness** — does the generated answer cover all required atomic claims? | `hallucination_check_points` |
| **Per-category accuracy** — does any architecture systematically fail on diagnosis vs treatment? | `question_type` |
| **Step-1 vs Step-2&3 robustness** — does any architecture overfit to one mode? | `meta_info` |

## File-level invariants

For `medqa_ragas_golden.jsonl` (the accepted-only output):

- Every row has `quality_status == "accepted"`.
- Every row has `audit_issues == []`.
- Every row's `correct_answer` substring appears (case-insensitive) in `reference_answer`.
- Every row's `gold_context` is non-empty.
- Every row's `gold_chunks[*].chunk_id` exists in `data/indices/chunks.parquet`.

The `medqa_ragas_golden_needs_review.jsonl` file uses the same schema but with non-empty `audit_issues` and/or `quality_status == "needs_review"`.
